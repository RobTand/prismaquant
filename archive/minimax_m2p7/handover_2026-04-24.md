## Correctness Notes

- INT2 and INT3 names should be canonicalized as `INT2`/`INT3`, not `NVINT2`/`NVINT3`, in user-facing PrismaQuant terminology. The vLLM patch still uses `nvint` internally as an implementation label.
- FP8 and BF16 source weights should pass through unquantized when the allocator selects source passthrough formats. Do not requantize source FP8 into MXFP8 unless explicitly selected.
- The MiniMax export uses mixed groups:
  - `pack-quantized`, 2-bit int, tensor group 16, fp8 scales.
  - `nvfp4-pack-quantized`, 4-bit float, tensor group 16, fp8 scales.
  - `float-quantized`, 8-bit float block.
  - `pack-quantized`, 3-bit int, tensor group 16, fp8 scales.
- The current serving baseline uses `NVINT_ACT_FORMAT=bf16`. Do not switch to the A4/NVFP4 activation path until A16 correctness and profiling are understood.

Key behavior in the current patches:

- Dense INT2/INT3 keeps checkpoint fp8 per-group scales resident.
- Dense INT2/INT3 uses compact bf16 row-wise `weight_global_real` instead of materializing huge bf16 combined-scale tensors.
- MoE INT2/INT3 keeps fp8 per-group scales resident and uses compact row-wise global vectors.
- MoE active expert loop was reduced from all experts to `torch.unique(topk_ids)`, which made startup/warmup viable.
- Triton INT2/INT3 kernels accept fp8 scale plus optional row-wise `global_real` and multiply the row scale inside the kernel.
- Memory guard kills the container before host RAM+swap hard-thrashes.

Known rough edges:

- `_apply_a4` in `compressed_tensors_w23a16_nvint.py` still references `weight_scale_bf16`; this is not exercised with `NVINT_ACT_FORMAT=bf16`.
- The patched vLLM worktree is dirty and includes unrelated/unreviewed files. Audit before committing.

## Performance Finding

Current performance is bad but coherent:

- Recent vLLM logs showed roughly `1-2 tok/s`.
- TTFT histogram was extremely poor, with `time_to_first_token_seconds_sum=213.67` over 3 completed requests.
- `EngineCore` ran hot on CPU while GPU SM utilization was high.
- `py-spy` exists in the container, but attaching to the running container failed because ptrace permissions were not granted.

The dominant suspected bottleneck is the custom MiniMax MoE INT2/INT3 path:

- `torch.unique(topk_ids).detach().cpu().tolist()` forces a GPU-to-CPU sync per MoE layer.
- Decode with top-k routing runs per-active-expert gate/up and down GEMMs separately.
- For MiniMax top-k around 8, this is about 16 tiny custom GEMMs per MoE layer per token.
- Across many layers this becomes hundreds to roughly a thousand tiny Triton launches per generated token.
- `--enforce-eager` prevents CUDA graph capture, so launch and Python overhead are paid every token.

Next optimization target: vectorize/group the MoE path before tuning individual GEMM configs. The priority is to remove CPU syncs and process routed experts in bulk.

```bash

  --cap-add SYS_PTRACE \
```
